### 三、 核心功能详解：如何实现 "Express Interest"？

这个功能是满足“用户间互动”和“数据持久化”的关键。你可以把它理解为**“轻量级的交换请求”**。

#### 1. 数据库逻辑 (Database Model)
你需要一个关联表（或模型）来记录这种关系。
*   **Table: `Interest`**
    *   `id`: 主键
    *   `sender_id`: 点击按钮的人 (User ID)
    *   `post_id`: 被点击的帖子 (Post ID)
    *   `timestamp`: 点击时间

#### 2. 用户操作流程 (User Flow)
*   **Step 1 (前端)**: 用户 A 浏览用户 B 的“Python 辅导”帖子，点击 `I'm Interested` 按钮。
*   **Step 2 (后端)**: 
    *   后端接收到请求，首先检查：用户 A 是否就是发帖人？是否已经点击过？
    *   在 `Interest` 表中存入一条：`sender_id=A, post_id=Python帖子的ID`。
*   **Step 3 (反馈)**: 按钮状态立即变为 `Requested` (禁用状态)，给用户即时反馈。
*   **Step 4 (信息披露 - 关键)**: 
    *   当用户 B 登录后，进入 **Dashboard -> Interests Received**。
    *   系统查询：`SELECT * FROM Interest WHERE post_id IN (用户 B 的所有帖子)`。
    *   用户 B 会看到：“用户 A 对你的 'Python 辅导' 感兴趣！他的邮箱是：A@student.uwa.edu.au”。
    *   *注：这种设计规避了复杂的即时聊天系统，但完成了“连接用户”的目标，非常符合 Agile 开发。*

Since you are a **4-person team**, I have broken down the project into four distinct modules. Each module has a specific set of Frontend pages, Backend logic, and "Core Technical Tasks." This ensures that every member has a clear area of responsibility for their GitHub contributions.

Here is the technical specification for **UWA Skill-Swap**.

---

### 1. Frontend Page List (The Views)

*   **`login.html` / `register.html`**: Clean, centered forms with validation messages (e.g., "Email must be a @student.uwa.edu.au address").
*   **`index.html` (Home)**: The "Skill Discovery" hub. Contains the Search bar, Category filter buttons, and a responsive Grid of Skill Cards.
*   **`post_detail.html`**: Expanded view of a skill. Shows full description, "Requested by" count, the Comment section, and the "Express Interest" button.
*   **`create_post.html` / `edit_post.html`**: Forms for title, category dropdown, and a long-form text area for the description.
*   **`dashboard.html`**: The private user area. Split into three sections (using Tabs or Cards):
    1.  *My Skills*: List of skills the user is offering (with Edit/Delete buttons).
    2.  *Incoming Interests*: A list of people who want to learn from the user (showing their names/emails).
    3.  *My Requests*: A list of skills the user has expressed interest in.

### 2. Functional Specification (English)

#### Module A: User Authentication & Profile Management
*   **Backend:**
    *   User Model: Store `id`, `username`, `email`, and `password_hash`.
    *   Security: Implement password hashing using `werkzeug.security`.
    *   Session Management: Use `Flask-Login` for login/logout and `@login_required` decorators.
*   **Frontend:**
    *   Build Login and Registration pages.
    *   Implement client-side form validation (e.g., password length, email format).
    *   Flash Messaging: Show success/error alerts (e.g., "Invalid credentials").

#### Module B: Skill Post Management (CRUD)
*   **Backend:**
    *   Post Model: Store `id`, `title`, `description`, `category`, `timestamp`, and `user_id` (foreign key).
    *   Routes: Create `POST` to save new skills and `GET` to retrieve them.
    *   Authorization: Ensure only the owner can `UPDATE` or `DELETE` their own posts.
*   **Frontend:**
    *   Build the Create/Edit Post forms.
    *   Design the "Skill Card" component for the homepage (showing snippets of info).
    *   Handle image placeholders or icons based on categories (e.g., a "Code" icon for Programming).

#### Module C: Discovery, Search & AJAX Filtering
*   **Backend:**
    *   Search Engine: Create a route that filters the database by keyword (title/description).
    *   **AJAX Endpoint**: Create a specific route (e.g., `/api/filter`) that returns Post data in JSON format based on the selected category.
*   **Frontend:**
    *   Implement the Search Bar functionality.
    *   **AJAX Logic (JQuery)**: Write a script so that when a user clicks a category (e.g., "Music"), the skill list updates instantly without refreshing the page.
    *   Build the responsive CSS layout for the Home page.

#### Module D: User Interaction (Comments & Interest System)
*   **Backend:**
    *   Interest Model: A "Junction Table" linking `Users` to `Posts` to track who is interested in what.
    *   Comment Model: Store text, timestamps, and links to `User` and `Post`.
    *   Dashboard Logic: Write complex queries to fetch "Who is interested in my posts" to display on the dashboard.
*   **Frontend:**
    *   Build the Comment Section UI on the detail page.
    *   Implement the "Express Interest" button (toggle state: "Interested" vs "Request Sent").
    *   Design the Dashboard tabs/layout to display interaction data clearly.
